## Context

项目目标为「投票小程序」：先网页端、后 App（见 `RESEARCH.md`）。当前变更仅解决**身份标识与会话**：用户通过**姓名 + 学号**登录、可登出；尚无服务端账号体系，属自用辅助场景。

## Goals / Non-Goals

**Goals:**

- 在客户端建立可复用的「当前用户」模型（姓名、学号），供后续上传作品、投票等能力读取。
- 登录前做清晰的前端校验；登出后状态一致、可再次登录。
- 刷新页面后仍保持登录态，直至用户主动登出（与提案中「持久化」一致）。

**Non-Goals:**

- 密码、二次验证、OAuth 等强鉴权。
- 服务端会话、JWT、学号全局唯一性校验、防伪造（自用场景接受本地可篡改风险）。
- 与「上传视频」「班级列表投票」的完整业务流程耦合（由后续变更接入）。

## Decisions

1. **登录态存储位置（首选 `localStorage`）**  
   - **选择**：将 `{ name, studentId }`（字段名实现时可用英文 key）序列化存入 `localStorage`，应用启动时读取并恢复登录态。  
   - **理由**：满足刷新后仍登录；实现成本低。  
   - **备选**：`sessionStorage`（关闭标签即丢）— 与「持久化」弱一致，故不作为默认。

2. **校验规则（与 spec 对齐，实现时照抄）**  
   - **姓名**：trim 后非空；长度上限 **32** 个 Unicode 码点（避免异常长输入）。  
   - **学号**：trim 后匹配 **`^[0-9A-Za-z]{4,20}$`**（兼容常见数字/字母学号；若学校规则不同，可在后续变更收紧/放宽并改 spec）。  
   - **理由**：可测、可本地化提示；避免无界字符串。

3. **UI 放置**  
   - **选择**：全局布局（如顶栏）展示当前用户姓名 +「登出」；未登录时展示「登录」或主导航跳转至登录页/弹层。  
   - **理由**：与后续各页共享同一身份上下文。具体路由结构随脚手架选定。

4. **无后端时的「账号」语义**  
   - **选择**：「账号」= 本机当前保存的姓名+学号组合；同一浏览器可切换用户（登出再登录）。  
   - **理由**：与提案一致，不为本变更引入假注册流程。

## Risks / Trade-offs

- **[风险] 本地存储可被用户或脚本篡改** → **缓解**：自用可接受；未来接后端时以服务端身份为准，客户端仅作展示与过渡。  
- **[风险] 学号格式假设与真实数据不符** → **缓解**：在 Open Questions 中记录，必要时单开变更调整正则。  
- **[权衡] 无密码** → 任何能操作设备的人可代登；符合非商业自用范围。

## Migration Plan

- 首版无线上迁移。若日后引入后端用户表：可增加「绑定远端账号」流程，并保留一段兼容期读取旧 `localStorage` 键或提示重新登录。

## Open Questions

- 学号是否允许含「-」或学校统一前缀（若需要，放宽正则并同步修改 spec）。  
- 是否在 UI 上完整展示学号，或仅展示后四位（隐私）；当前 spec 要求至少展示姓名，学号展示可随产品偏好微调。
