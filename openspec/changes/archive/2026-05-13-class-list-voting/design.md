## Context

`web/` 已具备：登录态（`user-login`）、IndexedDB 班级作品（`class-submissions`）及列表内联预览。本变更在**纯前端**场景下增加**投票**：与当前登录学号绑定的一票制，以及改投确认弹窗；不引入服务端计票。

## Goals / Non-Goals

**Goals:**

- 在班级列表中浏览他人作品（沿用既有内联预览；可在实现阶段增加「放大查看」等不改变票规的增强）。
- 落实**每人一票**（按当前登录用户的学号维度），改投需确认并原子切换选票。
- **禁止**给本人提交的作品投票（与「其他同学的作品」一致）。

**Non-Goals:**

- 防刷票、设备指纹、服务端审计与实时排行榜。
- 匿名投票、多选投票、加权票。
- 删除作品后的服务端级联（客户端见下方孤儿票策略）。

## Decisions

1. **选票持久化使用 `localStorage`**  
   - **选择**：单键 JSON，例如 `vote_app.cast_vote.v1` → `{ voterStudentId, votedSubmissionId }`。  
   - **理由**：体积极小、与登录态同栈；读取早于列表渲染即可完成高亮。  
   - **备选**：IndexedDB 单独 store — 过重。

2. **投票者与存储一致性**  
   - **选择**：恢复选票时 **MUST** 校验 `voterStudentId === useAuth().user.studentId`；不一致则**忽略**持久化内容（视为未投票），避免串号。  
   - **理由**：同一浏览器多人轮流登录的自用场景。

3. **改投确认交互**  
   - **选择**：浏览器原生 `window.confirm` **或** 自定义 Modal（实现任选），但必须为**阻断式确认**（用户必须显式选确认/取消）。  
   - **理由**：满足「弹窗提示」与无障碍可读性；自定义 Modal 更利于中文文案与样式统一。

4. **禁止自投**  
   - **选择**：当 `submission.uploaderStudentId === currentUser.studentId` 时隐藏投票按钮或置灰并展示「本人作品」提示。  
   - **理由**：与需求语义一致，减少无效提交。

5. **孤儿票**  
   - **选择**：列表加载完成后，若持久化中的 `votedSubmissionId` 不在当前 `listSubmissionsDesc()` 结果集中，则**清除**持久化选票并刷新 UI。  
   - **理由**：避免指向已删作品的幽灵票。

6. **预览范围**  
   - **选择**：首版不强制单独「预览页」；以内联预览 + 投票控件同卡展示为主。若实现放大预览，视为 UI 增强，不改变票规。

## Risks / Trade-offs

- **[风险] 本地票数可被篡改** → **缓解**：自用场景接受；接后端后以服务端为准。  
- **[风险] 多标签页并发改票** → **缓解**：以最后一次写入为准；可选 `storage` 事件同步（Open Question）。  
- **[权衡] `confirm` vs Modal** → **缓解**：spec 只约束确认/取消语义，不绑定具体组件 API。

## Migration Plan

- 首版无迁移。若未来接 API：将 `{ voterStudentId, votedSubmissionId }` 映射为 POST `/votes` 与 DELETE/PUT 语义。

## Open Questions

- 是否需要在多标签页间**实时同步**选票（`storage` 事件）。  
- 是否在首版加入**放大预览**（灯箱）以减轻长视频在列表中的占用。
