## Why

班级列表里大家只能看到「自己是否已投」，看不到每件作品**在全班的得票情况**，不利于公开、透明的评选氛围。要在每条作品上展示「当前票数」，必须先有**跨浏览器、可汇总**的投票数据源；目前选票只写在**本机 localStorage**，无法统计全班，因此需要把投票与计数打通后再做展示。

## What Changes

- 在班级列表的**每一条作品**上展示一个**非负整数**的「当前票数」（含 0），文案或样式上对用户可理解为「得票」或「票数」。
- 票数 MUST 反映**全班范围内**已生效的投票汇总（与仅当前设备可见的本地状态区分）；实现上需**服务端（或等价集中存储）**记录投票并支持按 `submissionId` 聚合计数。
- 用户投票、改投、删除作品导致选票失效等行为，须与既有 `class-voting` 规则衔接，并在聚合结果上体现（例如改投后旧目标票数减、新目标加；孤儿票清除不计入任何作品）。
- **非目标**：防刷票/强身份鉴权（与现上传、删除 API 同等信任模型即可在首版沿用）；实时毫秒级推送；按小时导出报表。

## Capabilities

### New Capabilities

- `vote-tally`: 服务端集中记录投票（按投票者身份去重、每人至多一票指向某一作品），并提供按作品汇总的得票数；供列表与前端展示使用。

### Modified Capabilities

- `class-voting`: 在班级列表 UI 上为每条作品展示当前票数；投票/改投须使服务端 tallies 与既有「一人一投、改投确认、不可投本人」等规则一致（持久化从纯 localStorage 扩展或迁移为与服务端同步，具体见 `design.md`）。

## Impact

- **API（`api/server.mjs` 或独立模块）**：新增或扩展投票写入与读计数接口；持久化形态（如独立 `votes.json` / 每提交目录旁索引）在 `design.md` 定稿。
- **前端（`web/`）**：列表拉取或合并票数；投票成功后刷新或增量更新计数；与 `voteStorage` 的协调（迁移、双写或替换策略）。
- **运维**：部署顺序与数据文件权限与现有 `VOTING_DATA_DIR` 一致；若有 Nginx 仅代理路径，一般无需改路由，除非新增独立 path。
